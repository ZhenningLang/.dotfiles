# Debug

## 1. 收集信息

如果 `$ARGUMENTS` 非空，将其作为 bug 描述的起点。否则交互式收集：

- 精确的错误信息 / 异常堆栈
- 复现步骤（或最小复现条件）
- 预期行为 vs 实际行为
- 最近的相关变更（`git log`）

不能复现的 bug 不要尝试修复——先找到复现路径。

## 2. 假设与分析

列出所有合理假设，按可能性排序：

| 层级 | 检查方向 |
|------|---------|
| 语法/类型 | 拼写、类型不匹配、import 错误 |
| 逻辑 | 条件反转、边界未处理、状态错误 |
| 集成 | 接口不匹配、数据格式变化 |
| 环境 | 配置、依赖版本、权限 |
| 并发 | 竞态、死锁、时序依赖 |
| 数据 | 脏数据、空值、编码问题 |

### 复杂 bug → 子 agent 隔离分析

当假设超过 3 个或涉及大量代码探索时，派发 researcher/worker droid 做隔离分析：

- 将收集的 bug 信息作为 prompt 传给子 agent
- 子 agent 在独立 context 中做假设排查（grep、读代码、git bisect）
- 子 agent 返回根因分析报告
- 主 session 基于报告执行修复

## 3. 定位

逐个验证假设，每个假设必须有证据支撑或排除：

- **二分法**：git bisect / 注释代码 / 日志注入
- **最小复现**：从完整场景逐步剥离到最小触发条件
- 确认根因后才进入修复

## 4. 修复

1. 先写捕获 bug 的失败测试
2. 最小修复（只改必要的，不顺手重构）
3. 跑通测试

## 5. 验证

- 原始 bug 已修复
- 回归测试通过
- 边界用例检查

## 禁止

- 猜测性修复（"试试这个看行不行"）
- 撒网式修复（同时改多处看哪个生效）
- 不复现就修
- 修复中夹带重构
- "改了好几处但不确定哪个修好了" → 停下来，回退，逐个验证
