# Plan

## 1. 理解需求

如果 `$ARGUMENTS` 非空，将其作为需求描述的起点。否则一次一个问题逐个深入：

- 要解决什么问题？当前痛点是什么？
- 预期行为是什么？边界条件？
- 不做什么？（显式排除项）
- 有没有已有方案可以参考？

### 规则

- 每次只问一个问题，等用户回答后再问下一个
- 提出 2-3 个方案并附取舍分析和推荐
- 遇到技术不确定点，建议用户触发 `/se-research`
- **硬门控：设计未经用户批准前，禁止写任何实现代码**

## 2. 产出 Spec

根据任务规模自适应粒度：

### 小任务（单文件/单函数级）

直接产出 Action 描述级 spec：

```
## 目标
[一句话]

## 步骤
1. [文件路径] — [操作] — [验证方式]
2. ...
```

### 复杂任务（多文件/多方向）

先产出整体 spec（Action 描述级），再按方向拆分为独立的详细 spec：

```
docs/specs/
├── overview.md          # 整体 spec：目标、架构、各方向概述
├── direction-a.md       # 方向 A 详细 spec（含完整代码）
├── direction-b.md       # 方向 B 详细 spec（含完整代码）
└── ...
```

每个方向的详细 spec 包含：
- 精确文件路径和操作
- 完整代码片段（不是伪代码）
- 每步的验证命令和预期输出
- 与其他方向的依赖关系

## 3. 自审

Spec 写完后执行检查清单：

- [ ] 无占位符（TBD/TODO/"后续完善"）
- [ ] 每步有具体文件路径
- [ ] 每步有验证方式
- [ ] 排除项显式列出
- [ ] 方向间依赖关系无环
- [ ] 垂直切片优先（按功能而非技术层拆分）

## 4. 用户审批

将 spec 提交用户审批。未批准不进入实现。

## 禁止

- 跳过需求理解直接写 spec（"需求很简单"也不行）
- spec 中留占位符或模糊描述
- 用"简化版/v1/先写死"偷偷缩减用户决策的范围
- 没有验证方式的步骤
