# Readable Metrics

## 1. 先定义“这组指标要回答什么问题”

在展示数据前，先明确：

- 这组指标要回答什么问题？
- 主要读者是谁？默认按工程师处理
- 用户要看的是现状、趋势、对比，还是异常定位？
- 哪些指标是真正关键指标？哪些只是辅助上下文？
- 是否有基线、阈值、SLO、环比/同比、预算线？

如果拿到的是一大坨原始数字，先分组，再决定怎么展示；不要直接 dump。

## 2. 指标分层

优先把指标组织成 3 层：

1. **结果层** — 最终 outcome（例如成功率、延迟、错误率、成本）
2. **驱动层** — 解释为什么变化（例如流量、缓存命中、队列堆积）
3. **动作层** — 接下来该盯什么、查什么、处理什么

单屏只展示最关键的 3–5 组指标。多了就分屏或分组。

## 3. 展示规则

在 Droid rich terminal 中，优先使用 `<json-render>` 做可视化；没有 rich terminal 时，退回 Markdown 表格/列表。

### 组件选择建议

| 场景 | 推荐组件 |
|------|---------|
| 一眼看状态 | `StatusLine` / `Badge` |
| 核心指标对比 | `Metric` / `KeyValue` |
| 多项数值比较 | `BarChart` |
| 时间趋势 | `Sparkline` |
| 明细与排序 | `Table` |
| 风险提示/说明 | `Callout` |

### 可视化约束

- 每个图只讲一个问题
- 图后必须跟一句解释：这说明什么
- 复杂图表必须有文字结论，不能让读者自己猜
- 非必要不要用过多颜色、过多面板、过多维度

## 4. 文案规则

- 先给一句话结论，再给图
- 数字必须带语义：基线、变化方向、异常程度、业务含义
- 非必要场景可适度取整，避免伪精确
- 专业缩写首次出现时解释
- 如果标题/注释/结论太硬太绕，调用 `/readable-rewrite` 或套用其规则
- 没有基线、阈值或对照时，不要硬做强对比；先明确“当前只知道现状”
- 没有证据时，不要把相关性写成因果

## 5. 默认输出结构

```md
## TL;DR
[一句话结论]

<json-render>...</json-render>

## What Changed
- [发生了什么]

## Why It Matters
- [为什么重要]

## What To Watch Next
- [下一步要盯什么]
```

如果没有 rich terminal，则退回：

```md
## TL;DR

## Key Metrics
| Metric | Value | Change | Meaning |
|--------|-------|--------|---------|

## What Changed
## Why It Matters
## What To Watch Next
```

## 6. 解释数字时必须补的上下文

至少补下面之一，最好补两项以上：

- 与上期比
- 与目标/SLO 比
- 与预算/阈值比
- 与同类项比
- 与用户体验/系统风险的关系

不要只说“错误率 2.3%”；要说“错误率 2.3%，高于 1% 阈值，且较上周翻倍”。

## 7. 自检

- [ ] 读者一眼能看到结论吗？
- [ ] 是否只展示了最关键的 3–5 组指标？
- [ ] 每个图后是否都有解释？
- [ ] 是否给了基线/阈值/对比，而不是裸数字？
- [ ] 是否告诉读者“接下来该看什么”？
- [ ] 终端布局是否可扫描，而不是 dashboard 噪音？

## 8. Gotchas

- 不要把“可视化”理解成“图越多越专业”
- 不要把辅助指标堆成主视图，掩盖核心结论
- 不要把需要解释的数字直接丢给读者自己解码
- 不要把时间趋势、分布对比、状态告警混在一个图里

## 关联技能

- 需要把标题、注释、结论改得更易读 → `/readable-rewrite`
- 需要把指标问题转成架构或实施方案 → `/think-plan`
- 指标反映出架构瓶颈，需要做结构层分析 → `/think-architecture`
