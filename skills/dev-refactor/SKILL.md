# Refactor

## 1. 确定范围

解析 `$ARGUMENTS`：

- `branch` 或分支名（如 `main`、`develop`）-> 比较当前分支与目标分支的差异
- `uncommitted` -> 重构当前未提交的变更（staged + unstaged）
- 文件路径、目录或符号名 -> 作为重构目标
- 留空 -> 询问用户选择上述模式

## 2. 分析

对 scope 内每个文件：

- 读代码，理解职责和调用关系
- Grep/Glob 搜索所有引用点（不凭记忆）
- 检查测试覆盖：关键路径无测试则先补
- 必要时先用 `/think-quality` 归因：问题到底是 Boundary、Locality、Convention、Explicitness、Testability 还是 Diff Purity

## 3. 输出重构计划（先判断再动手）

每条建议必须同时满足以下四个条件，否则不提：

1. **有明确 smell** -- 能指出具体问题，不是"感觉可以更好"
2. **收益 > 风险** -- 改完确实更好读或更好改
3. **范围可控** -- 影响的文件和调用者数量合理
4. **不需同时改行为** -- 若必须改行为才能重构，应拆成两步

按优先级分组输出：

- **P0/P1（应该改）**：结构掩盖 bug、命名误导、重复逻辑已导致分歧
- **P2/P3（可以改）**：可读性提升、命名微调、风格统一

每条包含：

- 文件路径 + 行范围
- 问题是什么
- 建议手法
- 为什么现在要改
- 改善哪些维度（Boundary / Locality / Convention / Explicitness / Testability / Diff Purity）
- 怎么验证收益与风险

### 确认方式

- P0/P1：逐条确认
- P2/P3：列为一组，用户选择全部执行 / 挑选 / 跳过

## 4. 执行

每次一个原子变更。改完立即验证。

### 铁律

- **行为不变** -- 只改结构，不改功能
- **先搜后改** -- 改任何符号前先 Grep 确认所有位置
- **可验证** -- 每步改完跑通测试

### 手法速查

| smell | 手法 |
|-------|------|
| 函数太长，有明显独立逻辑块 | 提取函数 |
| 同样的代码片段出现多次 | 提取共用模块，差异参数化 |
| 一堆 if-else/switch 做类似的事 | 数据驱动：配置表 + 查找 |
| 命名不清晰或前后不统一 | 全局重命名（含注释和 i18n） |
| 文件太长，职责混杂 | 拆分文件，保持 barrel export |
| 嵌套太深 | early return / guard clause |
| 函数只有一个调用点且无复用价值 | 内联消除 |
| 多余注释（复述代码、与代码不一致） | 删除或改写 |
| 冗余防御性代码（不可能触发的 try/catch） | 移除，保留必要的 |
| 死代码（未使用的 import/变量/函数） | 删除 |

### 每步自检

- 可读性变好了？密度高但不牺牲清晰度
- 数据集中了？配置/常量没有散落各处
- 命名统一？全局名详细、局部名精简
- 没引入不需要的抽象？（YAGNI）
- 耦合变松了？依赖方向合理？
- 是否真的让后续修改更局部、更显式、更易验证，而不是只是“看起来更优雅”？

## 5. 验证

自动探测项目的 typecheck / lint / test 命令（查 package.json scripts、Makefile、pyproject.toml 等），全部运行通过。

审查 `git diff`：检查遗漏的引用、意外删掉的逻辑。

最后问：重构后的代码比之前更易读、更易改吗？如果不是，回退。

## Gotchas

- 行为变更和重构要拆开；混在同一个 diff 里会让 review 和回归验证失真
- 不要因为“看着丑”就动手；先确认 smell、收益和影响面
- 命名、提取、删除死代码前都要先搜全局引用
- 如果改完后验证成本比之前更高，这通常不是好重构

## 禁止

- 重构中夹带功能变更
- 无测试覆盖下大范围重构
- 凭记忆改代码不搜索
- 为"优雅"引入不必要的抽象
- dirty working directory 上重构（先 commit/stash）

## 关联技能

- 需要先判断结构问题是什么 → `/think-quality`
- 重构完成 → `/guard-verify` 验证行为不变
- 重构后交付 → `/guard-review` → `/guard-ship`
- 连续失败 2 次 → `/think-unstuck`
