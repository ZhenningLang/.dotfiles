# 重构规范

## 1. 调研（改之前先搞清楚）

用 Grep/Glob 搜索待重构的符号，列出所有引用点。不凭记忆改代码。

对每个引用点：读周围代码，理解调用关系和数据流。

检查是否有相关测试覆盖。重构的基础是测试，关键路径无测试则先补。

输出一份影响清单：哪些文件、哪些函数、预估改动量。

## 2. 执行（小步快走）

每次只做一个原子变更。改完立即确认一致性（编译通过、引用完整）。

### 三条铁律

- **行为不变** -- 只改结构，不改功能
- **先搜后改** -- 改任何符号前先搜确认所有位置
- **可验证** -- 每步改完能跑通编译或测试

### 重构方向判断

改之前问自己代码哪里不对劲，对号入座：

| 问题 | 手法 |
|------|------|
| 函数太长，有明显独立逻辑块 | 提取函数 |
| 同样的代码片段出现多次 | 提取共用模块，差异参数化 |
| 一堆 if-else/switch 做类似的事 | 数据驱动：抽成配置表 + 查找 |
| 命名不清晰或前后不统一 | 全局重命名（包括注释和 i18n） |
| 文件太长，职责混杂 | 拆分文件，保持 barrel export |
| 嵌套太深，读着费劲 | early return / guard clause 扁平化 |
| 函数只有一个调用点且无复用价值 | 内联消除 |

### 改的时候对照检查

- 可读性有没有变好？代码密度高但不牺牲清晰度
- 数据是否集中？相关配置/常量有没有散落各处
- 命名统一吗？全局名详细、局部名精简
- 有没有引入不需要的抽象？（YAGNI）
- 耦合有没有变松？依赖方向是否合理？

## 3. ✅ 一定要做：验证（改完必须过）

```bash
# 编译/类型检查（按项目实际调整）
# TS: npx tsc -b --noEmit
# Go: go build ./...

# 格式化
# Go: gofmt -w
# JS/TS: 项目 lint 命令

# 运行测试
# 确保所有现有测试通过
```

审查 `git diff`：检查有没有遗漏的引用、意外删掉的逻辑。

最后问一句：重构后的代码比之前更易读、更易改吗？如果不是，回退重新想。

## ⚠️ 绝对不要做的事

- 不要在重构中夹带功能变更，一次只做一件事
- 不要在没有测试的情况下大范围重构
- 不要凭记忆改代码，永远先搜索
- 不要为了"优雅"引入不必要的抽象层
- 不要在 dirty working directory 上开始重构，先 commit 或 stash
