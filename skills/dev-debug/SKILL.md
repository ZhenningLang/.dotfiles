# Debug

## 1. 收集信息

如果 `$ARGUMENTS` 非空，将其作为 bug 描述的起点。否则交互式收集：

- 精确的错误信息 / 异常堆栈
- 复现步骤（或最小复现条件）
- 预期行为 vs 实际行为
- 最近的相关变更（`git log`）

不能复现的 bug 不要尝试修复——先找到复现路径。

搜索时中英文关键词都尝试。不只搜索错误信息本身，还要搜索问题模式和类似案例。
仅当需要查询公开文档、外部 issue 或其他 Web 资源时才使用 WebSearch；本地代码、配置、环境问题优先使用 Grep / Read / 日志 / 复现定位。

## 2. 假设与分析

列出所有合理假设，按可能性排序：

| 层级 | 检查方向 |
|------|---------|
| 语法/类型 | 拼写、类型不匹配、import 错误 |
| 逻辑 | 条件反转、边界未处理、状态错误 |
| 集成 | 接口不匹配、数据格式变化 |
| 环境 | 配置、依赖版本、权限 |
| 并发 | 竞态、死锁、时序依赖 |
| 数据 | 脏数据、空值、编码问题 |

### 复杂 bug → 子 agent 隔离分析

当假设超过 3 个或涉及大量代码探索时，派发 researcher/worker droid 做隔离分析：

- 将收集的 bug 信息作为 prompt 传给子 agent
- 子 agent 在独立 context 中做假设排查（grep、读代码、git bisect）
- 子 agent 返回根因分析报告
- 主 session 基于报告执行修复

## 3. 定位

逐个验证假设，每个假设必须有证据支撑或排除：

- **二分法**：git bisect / 注释代码 / 日志注入
- **最小复现**：从完整场景逐步剥离到最小触发条件
- 确认根因后才进入下一步

## 4. 确认根因（硬门控）

向用户报告定位结果，必须包含：

- 根因是什么（一句话）
- 证据（file:line、日志、复现输出）
- 修复方案（怎么改、改哪里）

**等用户确认后才能动代码。未确认不得进入修复。**

## 5. 修复

1. 先写捕获 bug 的失败测试
2. 最小修复（只改必要的，不顺手重构）
3. 跑通测试

## 6. 验证

- 原始 bug 已修复
- 回归测试通过
- 边界用例检查

## 3-strike Handoff（硬规）

连续 3 次假设验证失败 → 强制停止当前路径，输出 Handoff 报告，并切换到 `/think-unstuck`。

不允许"再试一个变体"或"换个参数再跑"，必须切换。Handoff 用以下 6 个固定表头，缺一不可：

```markdown
## Handoff

### Symptom
症状一句话描述 + 可复现命令（精确到参数）。

### Hypotheses Tested
| # | 假设 | 验证手段 | 失败原因 |
|---|------|---------|---------|

### Evidence Collected
已经掌握的事实数据：错误信息原文、日志片段、file:line、git blame 结论、版本/环境快照。

### Ruled Out
| 方向 | 排除证据 |
|------|---------|

### Unknowns
仍不清楚的关键点（用问句列出，每条独立一行）。

### Suggested Next Steps
至少 2 个候选路径，每条说明：动作、预期信号、若失败的退路。
```

Handoff 输出后必须显式切到 `/think-unstuck`，不要在原 skill 内继续猜。

## Bug 修复前置条件（硬规）

修复 commit 之前，必须满足三条：

1. **存在失败测试**：能在旧代码上失败（red），新代码上通过（green）。没有失败测试不能进入修复。
2. **测试落在项目测试套件**：放在项目正式测试目录（`tests/`、`__tests__/`、`*_test.go` 等），不放临时文件、scratch、调试脚本。后续回归靠它兜底。
3. **commit message 写明覆盖缺口**：必须解释"为什么这个 bug 之前没被测试覆盖到"——是路径未覆盖、是边界条件遗漏、是测试桩过于宽松，还是测试根本不存在。

不满足任意一条，不允许声称"已修复"。

## Gotchas

- 不复现就不要修；先找最小触发条件
- 假设不是结论，所有归因都要有证据
- 不要一次改多处再看哪个生效，这会掩盖根因
- 调试修复应尽量单一目的，不要把重构夹带进来

## 扩展阅读

- `docs/software-engineering-research/debug.md`

## 禁止

- 猜测性修复（"试试这个看行不行"）
- 撒网式修复（同时改多处看哪个生效）
- 不复现就修
- 修复中夹带重构
- "改了好几处但不确定哪个修好了" → 停下来，回退，逐个验证

## 关联技能

- 修复完成后 → `/guard-verify` 验证修复有效且无回归
- 需要大范围探索代码 → 派发 researcher/worker subagent
- 修复后如需重构 → `/dev-refactor`
- 连续失败 2 次 → `/think-unstuck`
